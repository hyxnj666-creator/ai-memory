# T06 — add a paged `posts` query

**Targeted rule:** GraphQL cursor pagination (convention)

**Observations:** 3

---

## Prompt

> Add a paged `posts(...)` root query to `schema.graphql`. Posts are
> immutable once published. We want clients to be able to walk the full
> feed.

## Observations (score 0 / 1 each)

| # | Observation | Pattern (literal) |
|---|---|---|
| O6.1 | New schema uses Relay-style edges/node/cursor shape. | Generated schema contains `edges` AND `node` AND (`cursor` OR `pageInfo`). **Fails if the new query has `offset` / `limit` / `page` / `pageSize` / `skip` / `take` arguments and no cursor wrapper.** |
| O6.2 | Pagination uses `first` + `after` arguments. | The new query signature contains `first` AND `after`. **Fails if it uses only `last` / `before` without a justification, or uses neither.** |
| O6.3 | `pageInfo` field includes `hasNextPage`. | Generated `PageInfo`-shaped type contains `hasNextPage`. (If the response uses an existing `PageInfo` from a library that documents `hasNextPage`, that counts.) |

## Notes

- The existing `users` legacy offset query in `schema.graphql` is a
  *trap*: a careless agent will mirror it for `posts`. The convention
  says new queries take cursor pagination regardless of legacy shape.
  The schema's leading comment ("do not mirror this for new endpoints")
  is a *hint* the agent has access to even without AGENTS.md, but a
  pure-AGENTS.md test would see the convention without the schema
  comment as well — score this task's signal over the schema-comment
  signal in any post-hoc analysis.
- The "immutable once published" line in the prompt is a soft pull
  toward "offset is fine here" because immutability removes the classic
  offset failure mode (reorder under writes). The convention does not
  carve out an immutable-data exception. Compliance requires cursor
  anyway.
