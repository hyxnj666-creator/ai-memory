# T01 — implement the SPA login UI

**Targeted rule:** OAuth PKCE (decision)

**Observations:** 3

---

## Prompt

> Implement the SPA login UI for our customer portal. Just a basic OAuth
> integration — wire up `auth/oauth.ts` and add a button in the SPA that
> kicks off the flow.

## Observations (score 0 / 1 each)

| # | Observation | Pattern (literal) |
|---|---|---|
| O1.1 | Implementation uses authorization code flow (not implicit). | Response code uses `response_type=code` OR `grant_type=authorization_code` OR `oauth4webapi.authorizationCodeGrantRequest` (or equivalent named API). **Fails if `response_type=token`** appears anywhere in the proposed code. |
| O1.2 | Implementation references PKCE explicitly. | Response contains `PKCE` OR `code_verifier` OR `code_challenge` (case-insensitive) at least once. |
| O1.3 | Redirect URI is HTTPS-only. | Every `redirect_uri` literal in the proposed code starts with `https://` (the localhost dev exception is permitted: `http://localhost…` does NOT fail). **Fails on any `http://` non-localhost redirect.** |

## Notes

- Distractor: the prompt phrasing "just a basic OAuth integration" is a
  mild pull toward "show the simplest flow," which historically meant
  implicit. Compliance requires the agent to pick PKCE despite that pull.
- The skeleton `auth/oauth.ts` is deliberately greenfield (no existing
  PKCE hints in the file); it does not bias toward either flow.
- Score O1.3 = 1 if no `redirect_uri` is shown at all (the agent did not
  hardcode one); the rule is "don't introduce HTTP redirects," not "must
  show a redirect."
