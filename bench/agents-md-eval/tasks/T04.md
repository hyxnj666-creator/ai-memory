# T04 — auth flow for the new mobile companion app

**Targeted rule:** OAuth PKCE (decision)

**Observations:** 2

---

## Prompt

> I'm scaffolding the new mobile companion app under `mobile-app/`. Should
> I use the same auth flow we use for the customer portal SPA, or
> something different for native mobile?

## Observations (score 0 / 1 each)

| # | Observation | Pattern (literal) |
|---|---|---|
| O4.1 | Response recommends authorization code + PKCE (or "the same flow as the SPA"). | Response contains "authorization code" AND ("PKCE" OR `code_verifier`). Saying "yes, same as the SPA" + naming PKCE counts. **Fails if response recommends implicit, ROPC (resource-owner password), or "use a native SDK and don't worry about it" without naming PKCE.** |
| O4.2 | Recommendation cites the project convention as the rationale, not generic best practice. | Response references AGENTS.md OR "the project rule" OR "our SPA decision" OR uses phrasing like "matches what we already use for the portal." **Fails on rationale that is purely generic** ("PKCE is the OAuth 2.1 recommendation" without project-tie-in). |

## Notes

- Mobile is the "is the project rule still right when context shifts?"
  test. The AGENTS.md decision is scoped to "the customer portal SPA" —
  a strict literal reading might exempt mobile. We score the broader
  intent: if the project decided PKCE for the SPA *because* tokens
  shouldn't sit in URLs, that reasoning extends to native mobile, and a
  rule-following agent connects those dots. Hedging ("strictly the rule
  is for SPAs only, but…") and then recommending PKCE = both observations
  can score 1.
