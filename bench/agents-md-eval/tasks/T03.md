# T03 — migrate legacy-portal off implicit flow

**Targeted rule:** OAuth PKCE (decision)

**Observations:** 3

---

## Prompt

> Migrate `legacy-portal/auth.ts` from its current flow to whatever we've
> decided on for new SPAs. Show me the rewritten file.

## Observations (score 0 / 1 each)

| # | Observation | Pattern (literal) |
|---|---|---|
| O3.1 | Agent identifies the target flow as "authorization code with PKCE" without asking the user "what flow?". | Response either rewrites the file directly using PKCE OR states "I'll migrate this to authorization code with PKCE" before any code. **Fails if the agent asks "which flow should I use?" or proposes implicit/hybrid as alternatives.** |
| O3.2 | Rewrite replaces `response_type=token` AND introduces a token-exchange step (`tokenUrl` / fetch to a `/token` endpoint with `code_verifier`). | Response code no longer contains `response_type=token`; new code includes a token-exchange call AND a `code_verifier` parameter. |
| O3.3 | Response acknowledges the deprecation/retirement aspect of the migration (this is the AGENTS.md "Retire the implicit-flow client config" / "deprecate the old client_id" line). | Response contains any of: "deprecate", "retire", "remove the legacy client", "remove `LEGACY_CLIENT_ID`", "phase out", or explicit guidance that the old config should be removed at end of quarter / after migration. **Fails on responses that quietly rewrite the flow without addressing what happens to the old client_id / config.** |

## Notes

- The skeleton `legacy-portal/auth.ts` is deliberately implicit-flow
  (`response_type=token` + URL fragment parser) so the agent has
  something concrete to migrate. The agent must (a) recognize this as
  "the deprecated flow we're moving off of" and (b) know what to migrate
  *to* — both signals come from AGENTS.md, not from the file itself.
- O3.1 fails on "which flow do you want?" because the AGENTS.md decision
  is the answer; asking is not honoring the rule.
