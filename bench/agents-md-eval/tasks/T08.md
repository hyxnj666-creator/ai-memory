# T08 — offset vs cursor for `comments`?

**Targeted rule:** GraphQL cursor pagination (convention)

**Observations:** 2

---

## Prompt

> We're adding a paged `comments(...)` root query. Should we use
> offset/limit or cursor pagination?

## Observations (score 0 / 1 each)

| # | Observation | Pattern (literal) |
|---|---|---|
| O8.1 | Response answers "cursor" without hedging. | Response contains "cursor" as the recommended choice, AND does NOT hedge with "depends on use case" / "either is fine" / "offset is simpler so consider it" without then resolving to cursor. **Fails on "either works, your call" or recommending offset.** |
| O8.2 | Response gives the project's actual reason (offset breaks under concurrent mutations) OR cites the project convention. | Response contains either: ("offset" AND ("mutation" OR "insert" OR "delete" OR "skip" OR "duplicat")) OR a reference to AGENTS.md / "the project rule" / "we agreed". |

## Notes

- This task directly asks the question the convention answers. A
  rule-honoring agent answers "cursor, because mutations cause offset to
  skip/duplicate rows" or cites AGENTS.md. A rule-ignoring agent answers
  "depends on your needs" — which is a perfectly fine generic answer
  for a general user but ignores that the project has already decided.
- The prompt deliberately phrases it as "should we use" rather than
  "implement cursor pagination," to give the agent the chance to
  default-to-offset if it's not consulting AGENTS.md.
