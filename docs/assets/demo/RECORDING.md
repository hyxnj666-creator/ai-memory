# Recording the hero GIF

`docs/assets/demo/demo.gif` is the GIF embedded near the top of `README.md`. It is **not** a screen recording — it is generated by [`vhs`](https://github.com/charmbracelet/vhs) from a checked-in script (`demo.tape`) and a hand-curated scenario (`scenario/`). Every render produces the same frames. There is no "I rerecorded it three times and they all looked different" problem to debug.

## Why vhs

Decision is logged in [`docs/launch-plan.md`](../../launch-plan.md#decision-log) (entries dated 2026-04-25). Short version:

- **Scripted** — the `.tape` file is the source of truth. Re-renders are deterministic.
- **Cross-platform** — `vhs` ships pre-built binaries for macOS / Linux / Windows, plus an official Docker image. Adding it to a maintainer's machine is one command.
- **Repo-friendly** — `.tape` files diff cleanly in PRs. Re-recording asciinema (or screen-capturing OBS) does not.

The demo also intentionally **does not run `ai-memory extract` live**, even though it is the headline command:

- Live `extract` needs an LLM API key in the render environment. We do not want to ship the demo behind an env-var dependency, and we definitely do not want to ship a GIF whose content depends on whatever model the maintainer happened to point at on render day.
- The artifact view (typed memory → `AGENTS.md`) is what actually persuades a viewer. The README copy + this file cover what `extract` does upstream — see [`README.md`](../../../README.md) and [`bench/cceb/README.md`](../../../bench/cceb/README.md) for the full extraction story.

If you ever change this trade-off (e.g. ship a separate "live extract" GIF), update [`docs/launch-plan.md`](../../launch-plan.md#decision-log) first so the decision history stays straight.

## Installing vhs

Pick the path that matches your OS. Any of these produces a `vhs` binary that the `npm run demo:render` script will find on `PATH`.

### macOS

```bash
brew install vhs
```

### Linux

```bash
# Debian / Ubuntu — see https://github.com/charmbracelet/vhs#installation
# for the apt repo + key. Quick path:
sudo mkdir -p /etc/apt/keyrings
curl -fsSL https://repo.charm.sh/apt/gpg.key | sudo gpg --dearmor -o /etc/apt/keyrings/charm.gpg
echo "deb [signed-by=/etc/apt/keyrings/charm.gpg] https://repo.charm.sh/apt/ * *" | \
  sudo tee /etc/apt/sources.list.d/charm.list
sudo apt update && sudo apt install vhs
```

```bash
# Or via Go (any distro with Go ≥ 1.20):
go install github.com/charmbracelet/vhs@latest
```

### Windows

`vhs` shells out to POSIX commands (`cat`, `ls`, `grep`, `rm`, `head`). Pure `cmd.exe` cannot run the demo — pick one of these instead:

```powershell
# Easiest: scoop installs both vhs and a POSIX userland.
scoop install vhs
```

```powershell
# Or: winget
winget install charmbracelet.vhs
```

```powershell
# Or: WSL — install vhs via the Linux instructions above, then run
# `npm run demo:render` from inside WSL with the repo mounted.
```

### Docker (any OS, no install)

```bash
# From the repo root:
docker run --rm -v "$PWD":/vhs ghcr.io/charmbracelet/vhs docs/assets/demo/demo.tape
```

The Docker image bundles `vhs` plus a Linux userland, so the same `.tape` script that works on macOS / Linux works here verbatim. This is the recommended path on Windows if you do not want to install scoop / winget / WSL.

### Verify

```bash
vhs --version
# vhs version vX.Y.Z (or similar)
```

## Rendering

```bash
npm run build           # ai-memory must be built — the demo invokes the CLI
npm run demo:render     # runs `vhs docs/assets/demo/demo.tape`
```

Render time is ~25–35 seconds (vhs steps through the tape at real speed). The output is written to `docs/assets/demo/demo.gif`.

The `demo:render` script intentionally does **not** rebuild — if you forget `npm run build`, you will render the previous binary. The first time you render after editing `.ai-memory/` parsing or rules logic, run `npm run build` first.

## Editing the demo

The demo has three layers; change the smallest one that fits your need.

### 1. Copy / pacing only

Edit `demo.tape`. The grammar is documented at <https://github.com/charmbracelet/vhs#vhs-command-reference>. Short reference:

| Directive            | Effect                                                           |
| -------------------- | ---------------------------------------------------------------- |
| `Type "..."`         | Types the string into the terminal.                              |
| `Enter`              | Presses Enter.                                                   |
| `Sleep <duration>`   | Pauses (`500ms`, `2s`, `2.5s`).                                  |
| `Hide` / `Show`      | Mutes / un-mutes the recording. Use for setup keystrokes.        |
| `Set TypingSpeed`    | ms-per-character. We use `35ms`.                                 |
| `Set FontSize`       | px. We use `14`.                                                 |
| `Set Width / Height` | Render dimensions in px.                                         |

Keep the total frame budget under ~30 seconds — longer GIFs are more likely to be skipped.

### 2. Memory content

Edit the markdown files under `scenario/.ai-memory/<author>/<type>/`. The file format is the same one `ai-memory` itself emits (see `src/store/memory-store.ts::renderMemoryFile`).

Two rules:

1. **Stay in English.** `scenario/.ai-memory/.config.json` pins `output.language: "en"` so AGENTS.md is HN-readable. If you switch to `zh`, expect to re-rewrite the surrounding README copy.
2. **Use LF line endings.** The parser was patched to handle CRLF, but tooling diffs are easier when staged content stays LF.

After editing memories, re-render (`npm run demo:render`) and visually verify both that `cat` frame 3 and `head` frame 5 still fit the panel.

### 3. The flow itself

Adding or removing a frame requires `.tape` edits **plus** a sanity check that the timing still adds up to ~30s. Use this rough budget:

```
typing_seconds = char_count * 0.035   # at TypingSpeed 35ms
total          = sum(typing_seconds + Sleep durations)
```

If you change the flow significantly (e.g. you decide to demo a different command than `rules --target agents-md`), update the corresponding decision-log entry in `docs/launch-plan.md` and re-evaluate whether the README copy still matches.

## Pre-commit checklist

Before committing `demo.gif`:

- [ ] `npm run build` succeeded.
- [ ] `npm test` passes (the demo depends on the parser; a regression elsewhere can break the GIF silently).
- [ ] `npm run demo:render` produced a fresh `demo.gif`.
- [ ] Open the GIF and confirm:
  - [ ] Every frame's text fits inside the 1180×720 panel.
  - [ ] The decision frame (frame 3) is readable for at least 4 seconds before the next frame.
  - [ ] No real path / username leaked into the recording (the scenario is fully synthetic, but the `Hide` block must come before `Show`).
- [ ] `git status` shows ONLY `docs/assets/demo/demo.gif` changed (no stray `scenario/AGENTS.md` — that file is gitignored, but double-check).

The GIF is a binary asset. Keep the diff to a single file per render and squash incidental re-renders.

## Troubleshooting

### `command not found: vhs`

`vhs` is not on `PATH`. See [Installing vhs](#installing-vhs). The npm script does not auto-install it.

### `npm run demo:render` finishes but the GIF is unchanged

Check `docs/assets/demo/demo.gif`'s mtime — vhs may have written it but your file watcher cached the old version. If the mtime is recent and the file size has not changed, your `.tape` is probably triggering an early exit. Run `vhs docs/assets/demo/demo.tape` directly and read the stderr.

### `head: command not found` (Windows cmd)

You are running vhs in `cmd.exe`. Re-run via Docker, scoop's POSIX userland, or WSL — see [Windows](#windows) above.

### `ai-memory: command not found` inside the GIF

The demo invokes `ai-memory` from `PATH`. The npm script does not run `npm link`, so `ai-memory` must already resolve. Either:

- run `npm link` once in the repo root before rendering, or
- edit `demo.tape` to invoke `node ../../../dist/index.js rules ...` instead.

### CRLF leak in the rendered AGENTS.md

If the rendered AGENTS.md shows `**Reasoning**:` / `**Alternatives**:` as raw text inside a rule body, the staged memory files were re-saved as CRLF (e.g. by an editor without `.editorconfig` honoured). The parser handles CRLF as of v2.4, but other downstream tooling (e.g. third-party parsers) may not. Re-save the files as LF.
