# Recording the hero GIF on Windows (manual fallback)

> **This is the fallback path.** [`RECORDING.md`](RECORDING.md) is canonical: the
> hero GIF should be re-renderable from `demo.tape` via `vhs` so every render
> is bit-for-bit identical and the script lives under version control. Use
> this file only when you cannot run `vhs` (no scoop / winget / WSL / Docker
> available, time-boxed pre-launch, etc.) and need to ship a GIF today.

If you can install Docker or run WSL, **stop and follow the canonical
[`RECORDING.md`](RECORDING.md) flow instead.** The maintainer's machine
producing inconsistent screen recordings is exactly the problem `vhs` was
introduced to solve.

---

## Why this file exists

`vhs` shells out to POSIX commands (`cat`, `ls`, `grep`, `rm`, `head`) and
relies on `go-rod` driving headless Chromium. On Windows native (no WSL, no
Docker), this combination has reproducible failure modes:

- Without scoop's POSIX userland on `PATH`, `vhs` exits early on the first
  POSIX command.
- With `HTTPS_PROXY` set, `go-rod` cannot reach its own localhost websocket
  and `vhs` panics with `use of closed network connection`.
- Even with the proxy cleared, headless Chromium on Windows can hang silently
  for minutes before producing output.

When the launch window matters more than render determinism, falling back to
manual screen capture against the same scenario unblocks shipping. This file
pins the recipe so the manual recording matches the `vhs` output as closely
as possible — same scenario, same dimensions, same 6-frame story.

---

## What you need

- **Recorder**: [ScreenToGif](https://www.screentogif.com/) (open source, free,
  has an integrated trim/limit/optimise editor). Install via scoop:

  ```powershell
  scoop install screentogif
  ```

  Alternatives: [LICEcap](https://www.cockos.com/licecap/) (lighter but no
  built-in editor), or Win+G → MP4 → convert with `ffmpeg -i in.mp4 out.gif`
  (most steps).
- **`ai-memory` on `PATH`**: from the repo root run `npm link` once.

  ```powershell
  cd D:\work\ai-memory
  npm run build
  npm link
  ai-memory --version    # verify it resolves and prints v2.4.0+
  ```
- **The scenario directory**: `docs/assets/demo/scenario/` is hand-curated and
  already in the repo. Do not modify the memory files for the recording —
  they are the same files `vhs` would see.

## Recording the 6 frames

The story is identical to [`demo.tape`](demo.tape). Treat that file as the
script: keep frame durations roughly the same so the rendered GIF feels like
a `vhs` re-render of the same scenario.

### 1. Stage the scene (do this BEFORE you start recording)

```powershell
cd D:\work\ai-memory\docs\assets\demo\scenario
Remove-Item AGENTS.md -Force -ErrorAction SilentlyContinue
Clear-Host
```

`AGENTS.md` is regenerated by frame 4. If it already exists from a previous
run the demo loses its punchline. The `.gitignore` keeps `scenario/AGENTS.md`
from being committed, but the file does need to be absent on disk before
frame 4 types `ai-memory rules --target agents-md`.

### 2. Start the ScreenToGif recorder

- Resize the recording frame to **1180 × 720 px** to match `Set Width 1180` /
  `Set Height 720` in `demo.tape`.
- Set frame rate to **15 fps** (15 is plenty for a typing-only demo and keeps
  the file under 2 MB).
- Position the frame over your PowerShell window so the prompt is visible at
  the top.

### 3. Type the 6 frames at a consistent pace

Each command is the PowerShell equivalent of the line in `demo.tape`. Type at
roughly the same speed `vhs` does (`TypingSpeed 35ms` ≈ 28 chars/sec — about
the rate of someone typing deliberately).

| Frame | Sleep | PowerShell command |
| ----- | ----- | ------------------ |
| 1 (framing) | 1 s | `# Extract typed decisions/conventions from your editor's chat history.` |
| 2 (layout)  | 2.5 s | `Get-ChildItem -Recurse .ai-memory -Filter *.md \| Resolve-Path -Relative` |
| 3 (decision) | 5.5 s | `Get-Content .ai-memory\conor\decisions\2026-04-22-adopt-oauth-pkce.md` |
| 4 (rules)   | 3 s | `ai-memory rules --target agents-md --all-authors` |
| 5 (artifact)| 4.5 s | `Get-Content AGENTS.md -TotalCount 25` |
| outro       | 1.8 s | `# Cursor, Claude Code, Windsurf and Copilot Chat all read AGENTS.md.` |

The sleep column is "how long you should pause **after** the command output
appears before typing the next command" — it gives the viewer time to read.
Total budget is ~30 s of typing + ~18 s of pauses.

The `cat` alias in PowerShell is `Get-Content`, so if you prefer the POSIX
visual you can type `cat .ai-memory\conor\decisions\2026-04-22-adopt-oauth-pkce.md`
in frame 3 — it produces identical output. `head` and `ls -1R` do **not**
alias by default; the table above uses the cmdlets that ship out of the box.

### 4. Stop the recorder, edit, export

In ScreenToGif's editor:

- Trim any pre-roll (the moment before frame 1 starts typing) and any
  post-roll (the moment after the outro pause ends).
- Optimise → "Reduce frame count" set to 50%, "Reduce colour palette" set to
  128 colours. Target file size is **< 2 MB** (GitHub allows 10 MB but smaller
  loads faster on README scroll).
- Export as `demo.gif`.

### 5. Drop it in the repo

```powershell
Copy-Item demo.gif D:\work\ai-memory\docs\assets\demo\demo.gif -Force
```

Then update both READMEs to swap the GIF placeholder comment for the actual
image markdown. The placeholder comments live near the top of each README and
already include the exact replacement string:

In `README.md` (English):

```markdown
![ai-memory in 30 seconds](docs/assets/demo/demo.gif)
```

In `README.zh-CN.md` (Chinese):

```markdown
![ai-memory 30 秒演示](docs/assets/demo/demo.gif)
```

Use the alt text exactly as written above so screen readers and the rendered
images on npmjs.com / GitHub stay consistent across the two language files.

## Pre-commit checklist

Before committing `demo.gif` — same checks as the `vhs` path, plus the
manual-recording-only ones:

- [ ] `npm run build` succeeded (the recording invokes the freshly built CLI
      via `npm link`).
- [ ] `npm test` passes (a parser regression elsewhere can break frame 4
      silently).
- [ ] The recording starts from a clean prompt — no shell history scrollback,
      no other window's output, no pre-existing `AGENTS.md`.
- [ ] The recording shows **only** the 6 frames listed above. No mistyped
      command, no `ll` / `dir` exploratory probes leaked into the recording.
- [ ] Open the rendered GIF and confirm:
  - [ ] Every frame's text fits inside the 1180×720 panel.
  - [ ] Frame 3 (the OAuth decision) is readable for at least 4 s before
        frame 4 begins.
  - [ ] No real path / username leaked into the recording — the prompt should
        say `…\demo\scenario>` and never `…\Users\<your-name>\…`.
  - [ ] File size is under 2 MB.
- [ ] `git status` shows ONLY `docs/assets/demo/demo.gif`, `README.md`, and
      `README.zh-CN.md` as changes (no stray `scenario/AGENTS.md`).

## When to switch back to vhs

The moment you can install `vhs` (e.g. you got access to a macOS/Linux/WSL
box, or set up Docker), re-render via `npm run demo:render`. The `vhs` GIF
should replace any manually-recorded one in the same commit so the repo
never has both. Reasons:

- Future demo edits should diff cleanly. `.tape` edits do, GIF re-recordings
  do not.
- The next person to touch the demo will have a much easier time if their
  baseline is "rerun the script", not "re-record the manual fallback".
- `RECORDING.md`'s troubleshooting section assumes `vhs` semantics.
  Reproducing a regression against a manually-recorded GIF wastes time.

If the manual fallback ever ships to npm / HN / a tagged release, log the
fact + the reason in [`docs/launch-plan.md`](../../launch-plan.md#decision-log)
so the next maintainer doesn't wonder why the GIF in `git log` does not match
a `vhs` re-render.
