## 1. Why this spike (the gap CCEB does not cover)

CCEB ([`bench/cceb/`](../bench/cceb)) measures **extraction quality**:
given a recorded chat, did `ai-memory extract` produce the right `convention
/ decision / architecture` records, with the right titles, the right
content, and the right de-duplication behaviour? It is the credibility lever
for "the LLM extractor isn't hallucinating." It is not — and was never
designed to be — a credibility lever for the *next* link in the chain:

> "OK the extractor works, but does the `AGENTS.md` you generate from
> those memories actually steer the agent toward those rules during a
> real coding session?"

This is a question every reader of the README, every HN thread on
"Cursor Rules / AGENTS.md don't actually work," and every prospective
adopter has a half-formed prior on, and we ship no number to disprove the
worst version of that prior. The same skepticism kept landing on
`AGENTS.md`-style projects during the v2.4 launch:

- "I've tried Cursor Rules / `AGENTS.md`. The agent reads them and ignores
  them half the time. Why is yours different?"
- "How do you know your generated rules steer the agent any better than
  random rules would?"
- "OpenAI / Cursor / Anthropic ship their own rule formats; you're a
  fifth wheel."

A defensible answer requires data we **don't currently have** anywhere in
the repo. v2.5-07 fills the gap with the cheapest possible experiment that
still produces a number we can put behind a sentence.

**One-sentence outcome we want to be able to publish:**

> "Across **N** controlled micro-tasks × 2 editors (Cursor + Claude Code),
> the rules generated by `ai-memory` were honored in **X / 50** observation
> points. Methodology, fixtures, and per-task scores are
> [bench/agents-md-eval/](../bench/agents-md-eval). Replicate it yourself
> in ~30 min."

The number being lower than 50 is **not a failure** — a number with
methodology beats no number. We commit to publishing whatever it lands
at.

## 2. Out of scope (what this experiment does NOT prove)

To stay defensible on HN we list what the number does **not** mean:

1. **Not a benchmark of Cursor or Claude Code's intrinsic rule-following.**
   A low score may reflect rule clarity, not editor capability. We will not
   draw editor-vs-editor conclusions in the writeup beyond "both editors
   were tested; per-editor breakdown is in the table."
2. **Not statistical significance.** n=10 tasks × ~2.5 obs avg × 2 editors
   = 50 obs is a sanity check, not a confidence interval. The publication
   template is required to call this out (§7).
3. **Not generalizable to all rule shapes.** We test the three rules
   bundled in the v2.5-02 demo scenario (OAuth PKCE convention, GraphQL
   cursor pagination convention, billing event-sourcing architecture
   decision). Adoption-relevant rules have similar shape ("MUST do X /
   reject Y") so it's a fair-but-not-universal sample. The writeup says
   so.
4. **Not a measurement of long-term behaviour drift.** We score one
   single-shot interaction per task. Rules that decay over multi-turn
   conversations are not exercised.
5. **Not blind.** The maintainer scoring the runs is the same person who
   designed the rubric. We mitigate with hard-coded
   "honored = these literal tokens appear / these tokens are absent"
   patterns wherever possible (§5) so a future contributor scoring the same
   run produces the same number ±1.

## 3. Methodology (locked)

### 3.1 The controlled repo

A single empty-ish directory under `bench/agents-md-eval/controlled-repo/`
containing:

- A frozen `AGENTS.md` (committed; **not** regenerated per run — see §4
  for why).
- Skeleton files the tasks reference (`schema.graphql`, `auth.ts`,
  `migrations/0001_init.sql`, etc.) so the agent has plausible context
  without leaking hints.
- No `.cursorrules`, no other rule files, no README mentioning the rules.
  AGENTS.md is the **only** signal of the bundled conventions.
- A `legacy-portal/` subdir with implicit-flow OAuth code, used by T3, so
  "migrate this" tasks have something to actually migrate.

The maintainer copies the controlled-repo into a scratch directory, opens
it in Cursor (then Claude Code), and runs the 10 tasks one at a time. The
controlled-repo itself is never modified during the run — diffs go in
`bench/agents-md-eval/results/<editor>/T0X.md`.

### 3.2 The frozen AGENTS.md

The AGENTS.md committed to `bench/agents-md-eval/controlled-repo/AGENTS.md`
is the verbatim output of the v2.5-02 demo scenario today (2026-04-27),
generated via the same code path the npm package ships:

```
node dist/index.js try --keep
# copy AGENTS.md from the printed tmp dir into the controlled-repo
```

The fixture is **frozen** rather than regenerated per run for two reasons:

1. **Reproducibility.** Future writer changes (column wrapping, section
   reordering, a new "Why" line) would silently invalidate prior scores.
   Pinning the literal bytes the agent sees lets us re-score later runs
   apples-to-apples.
2. **Fairness on HN.** A reviewer must be able to clone the repo and read
   the exact rule text the agent saw. They cannot do that if the rules
   are regenerated from a moving demo scenario.

`bench/agents-md-eval/scripts/verify-agents-md.ts` regenerates AGENTS.md
from the bundled demo scenario, normalizes both files (strips date headers
and absolute paths), and diffs them. It is run in CI: drift fails the
build with "the v2.5-02 writer changed; either re-spike v2.5-07 or restore
the previous writer."

### 3.3 The 10 micro-tasks

Each task lives at `bench/agents-md-eval/tasks/T0X.md` and has four
sections — copy/paste-able into either editor:

| Section | Content |
|---|---|
| **Prompt** | Verbatim message to send to the agent. Phrased as a natural dev request, not "follow the rule about PKCE." |
| **Targeted rule(s)** | Which AGENTS.md sections this task should activate (for cross-checking that the rule set we shipped is well-covered, not for the agent to see). |
| **Observations** | 2–3 0/1 yes-no checks the maintainer scores after the run. Each is phrased as a literal pattern check ("response mentions `code_verifier` OR `PKCE`") to keep scoring objective. |
| **Notes** | Known traps, distractor signals (e.g. T08 explicitly *asks* "should we use offset/limit or cursor"; the rule says cursor; agent must commit), expected confound modes. |

The 10 tasks split as:

- **5 tasks targeting the OAuth PKCE rule** (T01–T05)
- **4 tasks targeting the GraphQL cursor-pagination rule** (T06–T09)
- **1 cross-rule task** (T10) — a "design the new app" prompt that should
  surface both rules in a single response.

Total observations: 25 per editor × 2 editors = **50** — matching the
single round-number we publish. The split (5-4-0-1, 25 obs total) is
locked in §5.

> **Spike-first finding (2026-04-27): the bundled demo scenario has a
> third memory — billing event-sourcing — that is `type: architecture`,
> and the v2.5-02 AGENTS.md writer (`src/rules/agents-md-writer.ts:120-153`)
> deliberately filters to `convention + decision` only. Architecture-type
> memories never appear in the generated `AGENTS.md` the agent reads.**
>
> If we had blindly followed the obvious "3-3-3-1 split, one rule per
> bundled memory" plan, **30% of the experiment would have measured rule
> compliance against a rule the agent never saw** — producing systematic
> 0s on T07–T09 that we'd misinterpret as "the agent ignores
> event-sourcing rules" rather than "the AGENTS.md writer drops them on
> the floor."
>
> This is logged here as a writer-design question for v2.6+: should
> architecture-type decisions surface in AGENTS.md? Two reasonable
> answers:
>
> 1. **No** (current behaviour) — architecture decisions are
>    project-shape choices, not "while-coding" rules. AGENTS.md should
>    stay a tight rule list.
> 2. **Yes** — if a maintainer extracted "event-sourcing for billing,
>    `billing_events` is append-only" they almost certainly want the
>    agent to *not* generate UPDATE statements against `invoices`.
>    Filtering it out is data-loss-by-default.
>
> Both have merit; the answer is non-trivial and **out of scope for
> v2.5-07**. Logged as a known unknown in §8 with an explicit follow-up
> hook. The split this experiment locks (5-4-0-1) reflects what the
> shipped writer does *today*, not what it should do.

### 3.4 What "running" a task means

For each task, in each editor, the maintainer:

1. Resets the controlled-repo (fresh clone or `git clean -fdx`).
2. Confirms the agent has read AGENTS.md (Cursor: visible in @-mentions
   panel; Claude Code: shows up in `claude /context`).
3. Sends the prompt verbatim. **One** message; no follow-ups, no
   corrections. (Multi-turn drift is explicitly out of scope per §2.)
4. Captures the full response into
   `bench/agents-md-eval/results/<editor>/T0X-response.md`.
5. Scores each observation 0 or 1 in `scoring-template.csv` against the
   literal patterns specified in the task file. Ties (rule was implicit
   but not stated) score 0 — the rubric is intentionally strict to keep
   the "honored" claim defensible.

### 3.5 Editors and versions

| Editor | Version pinned at scoring time | Model |
|---|---|---|
| Cursor | latest stable as of run date (recorded in results) | Whatever the user's default Composer model is at run time |
| Claude Code | latest stable CLI as of run date | `claude-3.5-sonnet` (the public default) |

Both editor versions and both models are recorded in the results header so
later replications know what they're comparing against. We deliberately do
NOT pin a specific Cursor model — what we want to measure is "rules work
against the default agent the user gets out of the box," not "rules work
against a specific frozen LLM."

## 4. Fixture design principles (deliberate trade-offs)

| Principle | Reason |
|---|---|
| **Freeze AGENTS.md, don't regenerate** | See §3.2 — reproducibility + HN-defensibility. |
| **Empty controlled-repo, no other rule files** | Isolates the variable: the only thing steering the agent is the AGENTS.md content. A `.cursorrules` file in the repo would confound. |
| **Tasks phrased as natural dev requests** | "Add login to my SPA" is what real users type. "Apply rule §1 to my SPA" is what synthetic benchmarks measure. We want signal on the former. |
| **Distractor pull built into half the tasks** | T05 says "add pagination" without specifying flavor; T06 directly asks "offset or cursor?"; T08 asks for a refactor that *violates* the event-sourcing rule. Tasks where compliance is the path of least resistance produce inflated scores; we want at least 4/10 tasks where the agent has to actively notice the rule. |
| **Scoring patterns are literal** | "response contains `PKCE` OR `code_verifier`" is harder to argue with than "agent demonstrated understanding of PKCE." Compromises 30% of nuance to gain 100% of replicability. |
| **One-shot, single message** | Multi-turn capacity to recover from initial rule violations is a real feature of modern agents but is not what the rule format buys you. Single-shot isolates the rule's contribution. |
| **Same prompt across both editors verbatim** | Anything else (e.g. "tweak the prompt for Claude") would let us implicitly tune for the editor and inflate scores. |

## 6. Failure modes & re-spike triggers

We re-do the experiment (and re-publish a fresh number) when **any** of:

1. **The AGENTS.md writer output changes meaningfully** — `verify-agents-md.ts`
   detects this in CI. Trivial whitespace / date-line drift is normalised
   away; structural change (new section, reordered rules, different
   phrasing) re-triggers.
2. **The bundled demo scenario changes** (e.g. v2.6 ships a 4th
   bundled memory). The 10 tasks reference these three rules specifically;
   adding a rule means adding tasks.
3. **One of the targeted editors makes a default-behaviour change** that
   would invalidate the score (e.g. Cursor changes which rule files are
   read by default, or Claude Code changes how `AGENTS.md` is surfaced to
   the model).
4. **A reader submits a defensible edge case** that scores ≥1 differently
   from our score sheet. We add it to the rubric and re-run.
5. **Scores look implausibly good** (>45/50 on first run). The most likely
   explanation is that the tasks are too leading; we redesign rather than
   ship a number we cannot defend.

## 8. Known unknowns (recorded honestly)

These are gaps we can't close in v2.5-07 itself; they go in the writeup as
explicit threats to validity:

- **No real-world rule corpus.** We use the v2.5-02 demo scenario's three
  rules. A user with 50 conventions in their `.ai-memory/` may see
  different compliance — rule-set size and rule-overlap interactions are
  uninstrumented.
- **No model controlled.** Each editor's default model rolls forward; a
  re-run in 6 months may score very differently for reasons that have
  nothing to do with `ai-memory`.
- **One scorer.** The maintainer running it is also the project author.
  We mitigate with literal-pattern scoring (§5) but the right next step
  for v2.6 is "ship the fixture pack, ask one external contributor to
  re-score, publish both numbers."
- **No null baseline.** We do not run the same 10 tasks against an empty
  `AGENTS.md` to measure the lift over no-rules. Adding a baseline run
  doubles maintainer execution time; deferring to v2.6 is a deliberate
  scope choice. **If reviewers demand it, the fixture pack already
  supports it** — re-run with `AGENTS.md` empty and publish a paired
  delta.
- **The architecture-type writer-filter question** (§3.3 callout):
  whether the AGENTS.md writer's current `convention + decision` filter
  is correct, or whether architecture decisions like "billing is
  event-sourced" should also become rules. We take the path of measuring
  the writer's current behaviour rather than litigating its design in
  this PR; the question is logged as a v2.6 candidate in ROADMAP. If the
  v2.5-07 score is poor *and* a reviewer points out that one of our three
  bundled memories never reaches the agent, we re-spike the writer, not
  this experiment.
