# ADR 2026-04-24 — Naming & Rebrand Decision

**Status:** Accepted
**Date:** 2026-04-24
**Owner:** @hyxnj666-creator (Conor Liu)

## Context

After shipping v2.3.0, we debated rebranding the project as part of a 1000-star
push. Candidate names considered: **Engram**, **Mneme**, **Cairn**, **Axon**,
**Kairo**. We spent one working session auditing npm registry, GitHub, and
search volume for each.

## What we found (verified 2026-04-24, cached in [../competitive-landscape.md](../competitive-landscape.md))

Every name we considered is either (a) taken on npm by an abandoned "zombie"
package *and* (b) already used by at least one same-space MCP memory project
on GitHub:

| Candidate | npm owner | Same-space GitHub projects |
|---|---|---|
| engram | 2014 zombie (53 weekly downloads) | DeepSeek/Engram (4.3k★), tstockham96/engram (TS MCP for Cursor, 35★), engram-memory/engram, Clarit-AI/Engram, keggan-std/Engram |
| mneme | 2021 zombie (1 weekly download) | TheoV823/mneme, CVPaul/mneme, Methux/mnemo, mnemon-dev/mnemon, Memento |
| cairn | 2017 zombie (37 weekly downloads) | jasondostal/cairn-mcp, smcady/Cairn, grrowl/cairn |
| axon | — | harshkedia177/axon (646★, Feb 2026) — owns the name in this space |
| kairo | — | Close to AWS's new **Kiro** IDE → permanent spelling confusion |

We also discovered `alphaonedev/ai-memory-mcp` — a direct positioning competitor
(universal MCP memory, SQLite FTS5, "97.8% R@5 on LongMemEval") — confirming the
niche is crowded but not yet dominated.

## Decision

**Do not rename.** Keep:

- npm package: `ai-memory-cli`
- binary name: `ai-memory`
- GitHub repo: `hyxnj666-creator/ai-memory`

## Why

1. **No good name is actually free.** Every "memory / trace / marker" metaphor
   in English has a 1-5 project competitor already shipping. A rename wouldn't
   escape the crowded namespace — it would just add us to it.
2. **The real bottleneck is differentiation, not the name.** Our hero metaphor
   (`git diff`-able memory, conversation-to-rules pipeline, multi-source
   aggregation) is distinctive regardless of what we're called. pnpm, esbuild,
   vitest, turbo all thrived with descriptive names.
3. **Rename cost is real.** We already published `ai-memory-cli@2.3.0`. Every
   existing user's `package.json`, `.cursor/mcp.json`, CI scripts, and
   docs/blog SEO would break.
4. **Discovery is fine.** "ai memory mcp" literally matches our name — good
   SEO in this niche. The "generic" critique is aesthetic, not functional.

## Consequences

- Marketing and v2.4 engineering focus on differentiation, not branding
  (see [../launch-plan.md](../launch-plan.md)).
- README tagline rewritten to emphasize unique positioning: *"The AGENTS.md
  that writes itself — git-trackable knowledge from every AI coding session."*
- No domain purchase, no logo redesign, no npm name change.
- If this decision needs revisiting (e.g. a dominant `ai-memory-foo` fork
  appears), write a superseding ADR here — don't silently rebrand.

## Alternatives rejected

- **Rename to Engram/Mneme/Cairn** — direct conflict with established projects
  in our niche, would look like a copycat, SEO-worse than keeping current name.
- **Rename to a made-up word (Vercel/Tauri-style)** — solves uniqueness but
  costs 1-2 weeks of naming paralysis for a project whose moat is product
  differentiation, not brand.
- **Add a codename ("ai-memory / codename Lineage")** — looks try-hard,
  confuses users about what to `npm install`.

## Non-goals

- We are **not** promising never to rename. If a legal or trademark conflict
  arises, or if someone ships a dominant `ai-memory-v2` fork, reopen this ADR.
- We are **not** changing the binary from `ai-memory` to anything else.
