**Status:** Accepted

## Context

`ai-memory-cli@2.4.0` published to npm on 2026-04-26 (commit `3f21251`, tag `v2.4.0`). At publish:

- **Tests:** 431 across 25 files; typecheck clean; build clean.
- **Headline numbers:** CCEB v1 baseline F1 56.0% (P 43.8% / R 77.8%) on `gpt-4o-mini`, 9 fixtures.
- **Capabilities shipped this cycle:** AGENTS.md reverse-sync, `recall` git time-travel, `doctor` health check, dashboard polish (onclick escape + pagination at 50/page), CCEB framework + first published baseline, Cursor path display fix.
- **Outstanding from v2.4 plan:** hero GIF render (POSIX-only via `vhs`, deferred to a macOS/Linux machine; manual Windows fallback documented).
- **HN promotion:** decoupled from publish. Earliest target window Tue 2026-04-28 08:30 ET (= Beijing 20:30) per [`launch-plan.md`](../launch-plan.md).

The ADR [2026-04-25-category-positioning.md](2026-04-25-category-positioning.md) closed the "what slot do we claim" question. With v2.4 now in users' hands, the open questions are:

1. **What does v2.5 ship**, given the four "only we do this" claims have all landed code (#1 chat-history extraction, #2 AGENTS.md sync, #3 typed memory, #4 team-native)?
2. **Which 2026-Q2 market signals not yet in `competitive-landscape.md`** are worth chasing this cycle vs. parking for v2.6+?
3. **What's the single-biggest leverage** we have on the 30-day-after-launch metrics in [`launch-plan.md`](../launch-plan.md) (300-star floor, 1000-star stretch)?

## Decision

v2.5 scope is the following ten items, **stack-ranked by edge-per-day toward the 1000★ target**, not by feature size or technical interest. Items 01–03 are committed (must ship). Items 04–07 are strongly recommended, conditional on 01–03 landing on time. Items 08–10 are exploratory.

### Must ship (committed, week 1)

#### v2.5-01 — Lift CCEB precision from 43.8% to ≥60%, republish baseline

> **Update 2026-04-26 (post-ship):** Landed in development the same day this ADR was written. F1 56.0% → **76.2%**, Precision 43.8% → **66.7%**, Recall 77.8% → **88.9%** — both KPI floors cleared with margin (target was F1 ≥ 65% / P ≥ 60% / R ≥ 75%) and recall actually went up rather than paying for the precision lift. Implementation matches plan: prompt-side rewrite (no scorer / fixture / test changes). Cosmetic `openai (default)` model-label artefact also fixed in the same round. Four FPs and one FN remain — all four FPs are smaller-magnitude versions of the same over-extraction pattern, pointing at v2.5+ post-extract pairwise dedup as the next lever; the FN is `cceb-002` (convention-as-decision type-classification miss). New baseline + per-fixture deltas in [`docs/benchmarks/cceb-baseline.md`](../benchmarks/cceb-baseline.md); v2.4 numbers retained as historical context. Released as part of v2.5.0 (will date-stamp on publish).

The over-extraction pattern is well-understood — see [`docs/benchmarks/cceb-baseline.md`](../benchmarks/cceb-baseline.md) §"Concrete v2.5 work this baseline points at". Three concrete changes:

1. **Prompt-side:** explicit "one decision per discussion thread" instruction; merge sub-claims into the parent's `reasoning` / `alternatives` / `impact` rather than as standalone memories.
2. **TODO discipline:** require explicit commitment language ("we will", "TODO:", "action item") before promoting a follow-up action to a standalone TODO memory. Default conservative; user can override per-extract.
3. **Type-classification calibration:** targeted few-shot for the convention-vs-decision boundary (cceb-002) and architecture-vs-decision boundary (cceb-003).

Re-run baseline against `gpt-4o-mini` on the same 9-fixture suite. Target: **F1 65%+ at recall ≥75%** — don't trade recall below ~75% even for a higher F1 headline; recall regressions are silent failures users won't catch.

**Why this is item #1:** the republished number is the single most reusable piece of v2.5 launch content. F1 56% → 70% is a *re-postable* HN narrative ("we listened, here's the new number") — one of the few ways to legitimately get a second front-page window. It's also the bottleneck for AGENTS.md output quality: if `extract` precision is 44%, the auto-generated `AGENTS.md` is half noise.

While we're in there, fix the cosmetic `openai (default)` model-label artefact called out in the baseline honesty notes.

**Est:** 2–3 dev days. **Blocking:** nothing.

### Strongly recommended (target: ship if 01–03 land on time, weeks 2–3)

#### v2.5-04 — Anthropic Skills output (`rules --target skills`)

Anthropic Skills (introduced 2026-Q1) sit alongside `AGENTS.md` as the emerging cross-tool agent-instruction layer. They differ from `AGENTS.md` in two ways:

- Loaded **dynamically** by description matching, not always-on.
- Per-skill granularity (one file per capability) vs `AGENTS.md`'s single project-level rules document.

Natural mapping for `ai-memory`:

- One skill per (memory type × theme), e.g. `coding-conventions.skill`, `system-architecture.skill`, `decision-log.skill`.
- `description` field auto-generated from the constituent memory titles (concise "when to load" hint Claude uses to decide whether to pull the skill into context).
- Body composed from the relevant typed memories.

**Spike before commit:** validate the exact Skills schema against current Anthropic docs (the field set has been moving through 2026-Q1/Q2). Land a 1-page "Skills schema snapshot — date stamp" doc in `docs/` before implementing so future maintainers know which spec we targeted.

**Why fourth, not first:** `AGENTS.md` is the proven standard (60K+ adopters, Linux Foundation stewardship); Skills is a bet on the next 12 months. We should ship the bet, but only after the proven channel is humming. Also: Skills schema is still in flux; over-investing before the spec stabilises is wasted work.

**Why ship at all:** no competitor in any of our three buckets currently outputs Skills. Being first in this slot is the same play that worked for `AGENTS.md` reverse-sync in v2.4 — claim the unclaimed surface before the niche consolidates.

**Est:** 2–3 dev days (1 spike + 1–2 impl). **Blocking:** nothing technical; conditional on Skills spec being stable enough to target.

#### v2.5-05 — `--redact` flag for PII / secrets / internal URLs

Today `extract` sends conversation excerpts to the configured LLM provider. "Local-first" applies to *storage*, not to *extraction calls* — and the README is honest about this. A `--redact` flag (regex denylist for API keys, internal-domain URLs, email addresses) opens the "can I run this on a corp repo" door without changing the architecture.

Default rules to ship:

- `(sk|pk|aws|github_pat|ghp|xoxb)_[A-Za-z0-9]{20,}` — common token shapes.
- `https?://[^/]*\.(?:internal|corp|local|lan)/` — internal hostnames.
- RFC5322 emails.
- Optional: JWT-shaped tokens.

Custom rules via `.ai-memory/.config.json` `redact` array (regex strings; precompiled once per extract run).

**Why:** opens enterprise + safety-conscious users (HN niche, r/selfhosted, r/cscareerquestions). Cheap to ship, high signal-to-effort.

**Est:** 1 dev day. **Blocking:** nothing.

> **Update 2026-04-26 (post-ship):** Shipped same day, immediately after the v2.5-04 audit pass. Spike-first discipline held a second time: [`docs/redaction-policy-2026-04-26.md`](../redaction-policy-2026-04-26.md) was written *before* a single line of `src/extractor/redact.ts` landed, and the doc is now the executable spec — locked threat model (defense-in-depth, **not** a substitute for proper secret management; explicit out-of-scope list for image attachments / structured-PII vault inspection / retroactive scrubbing of existing memories), 10 default-on rules, 2 opt-in rules, opt-in-by-default decision (flipping default ON in a minor would silently change extraction quality for existing users; deferred to v3.0 with deprecation-warning hook in v2.5-09), audit-trail surface, and the explicit impl contract. The spike paid back during impl: it surfaced one ordering bug that would have shipped silently — the openai-key regex `sk-(?:proj-)?[A-Za-z0-9_-]{20,}` greedily consumes `sk-ant-api03-...` because both keys start with `sk-` and `-` is in the openai char class, so `anthropic-key` MUST run first in `DEFAULT_RULES`. Caught by a unit test, fixed in the same commit, footnoted in the spike doc as an "implementation order note". This is now a confirmed **two-for-two pattern** (v2.5-04 Skills schema snapshot + v2.5-05 redaction policy) — both spikes caught a real bug or wrong default before code landed. Officially adopted as the v2.5+ default approach for any work touching an external boundary (LLM call site, marketplace spec, agent-instruction format, etc.).
>
> Two design calls worth flagging vs the original sketch above:
> 1. **Default rules expanded from the sketch's 4 patterns to 10.** The sketch lumped `(sk|pk|aws|github_pat|ghp|xoxb)_` into a single regex; the actual implementation splits this into per-vendor patterns (`anthropic-key` / `openai-key` / `aws-access-key` / `github-pat` / `github-app-token` / `slack-bot-token` / `gcp-api-key` / `stripe-key`) so the audit trail shows *which* vendor's key was scrubbed (`[redacted: 2 openai-key, 1 stripe-key]` is more useful than `[redacted: 3 token]`). Stripe and GCP weren't in the sketch at all; added during the spike when surveying secret-shape catalogs. The internal-hostname pattern picked up `*.intra` as a fifth TLD beyond the sketch's 4 (still seen in some legacy enterprise networks).
> 2. **Two rules deliberately moved to opt-in.** The sketch listed JWT as "optional"; the spike added `aws-secret-key` to the same opt-in tier. Both have high false-positive rates against conversational prose (long base64 strings — auth tokens that aren't JWTs, base64-encoded image fragments, hash digests in build logs); flipping them on by default would produce noise that the audit trail can't distinguish from real hits. Available via `redact.enableOptional: ["jwt", "aws-secret-key"]` for users who want them.
>
> 51 redact unit tests + 4 CLI parse tests = `+55` vs v2.5-04 baseline of 465, **total 520 tests**. Built `dist/index.js` smoke-tested via `--help` (both flags surface in `Extract options:` and `Summary options:`). End-to-end LLM-call verification deferred to a follow-up RELEASE-CHECKLIST step — low risk because the redaction module is pure-functional and unit-tested 51 ways across catalogue invariants / golden inputs per rule / FP guards / opt-in toggles / config merging / validation + ReDoS guards / idempotency / `shouldRedact` precedence / `formatAuditTrail`. Estimated 1 dev day; actual ~3-4 hours including spike, impl, tests, docs.
>
> Strategic value: privacy / compliance posture is the missing leg of the "local-first" pitch. Until v2.5-05, "we never upload your memories" was true at the storage layer but glossed over the extraction-call layer. With `--redact`, the pitch becomes honest: opt-in scrub of secrets / PII / internal hostnames before the LLM call, with an audit trail that makes the trade-off transparent. This unblocks the `r/selfhosted` / HN-niche / "can I run this on a corp repo" segments that were watching v2.4 but unwilling to point it at real chat history. Compounds with v2.5-04 Skills (now part of the same release) and v2.5-03 submissions (the marketplace listings can now claim "redaction-aware extraction" alongside the three output targets).

> 1. **`--json` output ambiguity (real, MEDIUM).** When `--redact` ran on a clean conversation (no matches), the JSON output dropped the `redactions` field entirely — making "redacted, found nothing" indistinguishable from "redaction was never on". Spike-first discipline held a third time in succession: [`docs/codex-session-snapshot-2026-04-26.md`](../codex-session-snapshot-2026-04-26.md) was written *before* a single line of `src/sources/codex.ts` landed. > **Same-day audit pass (2026-04-26).** A re-read of v2.5-06 with fresh eyes surfaced 4 issues, all closed in the same wave — same shape as the v2.5-05 audit (4 issues) and the v2.5-04 audit (2 issues). Three of the four were real code bugs that would have silently broken downstream flows for users with Codex installed; one was active-marketing doc drift:
>
> - **Finding A — Bundle import whitelist out of sync (HIGH, real bug).** `bundle.ts:VALID_SOURCE_TYPES` is a hardcoded whitelist that `parseBundle()` validates `sourceType` against. The widening of `SourceType` to include `"codex"` in v2.5-06 didn't update this whitelist, so any bundle exported with a codex memory would throw `BundleParseError: sourceType must be one of: cursor, claude-code, windsurf, copilot` on import to another machine. Silent cross-machine portability breakage — the most-load-bearing of the four findings because there's no easy workaround. Fix widened the list + added an inline lock-step warning comment + a new test that exercises every production source type.
> - **Finding B — `watch` never fs.watch'd codex (HIGH, real bug, 30s latency regression).** `watch.ts` had an inline `cursor || claude-code` check for the fs.watch eligibility list. Codex's layout is structurally identical to Claude Code (recursive JSONL scan), so it should have qualified — instead it fell through to the 30s polling fallback for no technical reason. Fix promoted the inline check to a named `supportsFsWatch` helper with a docstring explaining the JSONL-vs-SQLite/JSON-blob rationale.
> - **Finding C — Missing config surface for `codex` (MEDIUM).** `AiMemoryConfig.sources` didn't have a `codex: { enabled: boolean }` slot, so privacy-conscious users had no documented way to opt out of codex monitoring. The `watch.ts` enabled-filter mirrored this gap with a 4-source if-ladder + `return true` fallthrough — indistinguishable at the call site from "I forgot to add this source". Fix added the field to both type + DEFAULT_CONFIG, and replaced the if-ladder with a named `isSourceEnabledInConfig` helper backed by an exhaustive `switch` over `SourceType` (TS compile error if a 6th source ships and the helper isn't updated).
> - **Finding D — Doc drift in 6 active marketing surfaces (LOW, cumulative).** RELEASE-CHECKLIST, ARCHITECTURE.md, competitive-landscape.md (5 spots), category-positioning ADR, submission-packet.md (8 spots), and README's §"Native AGENTS.md output" all still said "4 editors" / "Cursor / Claude / Windsurf / Copilot". Submission packet is consumed *after* `v2.5.0` ships (which includes v2.5-06), so 5-source claims are accurate at submission time. Historical surfaces (v2.4 release-notes section of CHANGELOG, v2.4 GIF outro, v2.5-06 ship-paragraph itself describing "joins the other four") deliberately left as historical record.
>
> Audit added **+8 tests** (1 bundle whitelist-coverage test + 7 watch source-routing tests in a new `watch-source-routing.test.ts` file); **total suite 556** (was 548 before audit). No production rule logic changed beyond the three documented bug fixes. Audit pass took ~45 min including this writeup.
>
> **The audit-pass pattern is now load-bearing.** v2.5-04 closed 2 issues, v2.5-05 closed 4, v2.5-06 closes 4 — all caught by re-reading the same artefacts with explicit "what would I test if I'd never seen this" framing. Cost per pass: ~30-60 min. Net signal per pass: 1-4 real bugs caught before they reach a user, plus the institutional memory the audit-fix tests encode (any future PR that misses one of the surfaces these tests pin will fail loudly in CI rather than silently in a user's bundle import / watch session / config file). Pattern is now mandatory for any feature touching an external-boundary surface (LLM call site / marketplace spec / agent-instruction format / **editor source adapter**).

#### v2.5-07 — AGENTS.md downstream evaluation

CCEB measures **extraction quality** (chat → typed memory). We have no number for **rule-following quality** (memory → AGENTS.md → does the agent actually obey?). Methodology:

1. Drop an `ai-memory`-generated `AGENTS.md` into a controlled empty repo seeded from the bundled v2.5-02 demo scenario.
2. Define 10 micro-tasks that exercise specific rules (e.g. "add a function that fetches data from `/api/users`" against a "use cursor pagination, never offset" convention).
3. Run each task in Cursor and Claude Code; manually score "rule followed" (0/1) per observation.
4. Publish a single number: "X / 50 rule observations honored across Cursor + Claude Code."

This is a **complementary** credibility lever to CCEB — different layer, different question. CCEB shields against "extraction is unmeasured"; this shields against "the rules you generate are noise the agent ignores".

**Why this matters for HN:** the most common skeptical comment on tools in this niche is "AGENTS.md / Cursor Rules don't actually steer agents reliably". Having a number — even a small-N one — turns a vibe argument into a data argument.

**Est:** 1–2 dev days. **Blocking:** nothing technical; methodology sign-off needed before publishing.

> **2026-04-27 update — agent-side prep complete; awaiting maintainer execution (status discipline mirrors v2.5-03).** Spike doc + fixture pack + drift guard + scoring template + publish-ready report skeleton are all committed: spike at [`docs/agents-md-eval-spike-2026-04-27.md`](../agents-md-eval-spike-2026-04-27.md); 10 micro-tasks at [`bench/agents-md-eval/tasks/T01.md`](../../bench/agents-md-eval/tasks/T01.md)…`T10.md`; frozen `AGENTS.md` fixture at `bench/agents-md-eval/controlled-repo/AGENTS.md`; CI-runnable `verify-agents-md.ts` drift guard; `scoring-template.csv` (50 rows, 25 obs/editor × 2 editors); results template at [`docs/agents-md-eval-results.md`](../agents-md-eval-results.md). The locked task split is **5 PKCE + 4 cursor-pagination + 1 cross-rule = 10 tasks / 25 obs / editor / 50 total**. Total test suite remains 556 (no production code touched).
>
> **One spike-first finding fed back, not silently buried.** The original ROADMAP wording assumed "5 known conventions" in the controlled repo; the actual writer (`src/rules/agents-md-writer.ts:120`) emits **only `convention + decision`** memories, dropping `architecture`. The bundled demo scenario has 3 memories (1 convention + 1 decision + 1 architecture), so AGENTS.md ships with **2 rules**, not 3 and not 5. A naïve "3-3-3-1 split, one rule per bundled memory" task plan would have spent 30% of the experiment scoring agents against a rule they could not see, producing systematic 0s on T07–T09 we'd misinterpret as rule-following failures. Whether `architecture`-type memories *should* surface in AGENTS.md (current writer says no — architecture decisions are project-shape choices, not while-coding rules; counter-argument: dropping "billing is event-sourced, no UPDATE on `billing_events`" is data-loss-by-default) is logged as a v2.6 candidate and explicitly out of scope for v2.5-07.
>
> **What still needs to happen for the ROADMAP `[x]`:** maintainer copies `controlled-repo/` to a scratch dir, opens it in Cursor + Claude Code, runs 10 prompts × 2 editors verbatim, scores 50 observations into `bench/agents-md-eval/results/scores.csv`, fills in the headline `X / 50` number into `docs/agents-md-eval-results.md`, propagates that number into README / competitive-landscape / submission-packet / CHANGELOG, then ships. ~30 min/editor + 15 min scoring + 5 min publishing per [`bench/agents-md-eval/README.md`](../../bench/agents-md-eval/README.md). Same status discipline as v2.5-03: agent-side prep is half the work, maintainer-side execution is the other half, and we don't claim ✓ until both halves are done.

#### v2.5-09 — README FAQ: "Doesn't 1M-token context obsolete you?"

A single section, 1–2 paragraphs, pre-empting an HN question we know will come:

- At ~$3/1M input tokens, dumping 1M tokens of raw chat history per query is ≈$3/query — token economy *gets harder*, not easier, as context grows.
- Long context degrades on retrieval at the tail (well-documented needle-in-haystack failures); compressed typed memory outperforms 1M raw tokens on the dimensions we measure.
- `AGENTS.md` is loaded by **every** editor session, not one query — per-token cost compounds across sessions.

**Est:** 0.5 dev day. **Blocking:** nothing.

> **Update 2026-04-27 (post-ship):** Shipped same day, immediately after the v2.5-08 prep close-out. Spike-first discipline held a sixth consecutive time (after v2.5-04 / v2.5-05 / v2.5-06 / v2.5-07 / v2.5-08): [`docs/1m-context-faq-spike-2026-04-27.md`](../1m-context-faq-spike-2026-04-27.md) locked claims + sources + placement + length cap + 4 re-spike triggers BEFORE any README copy was drafted. The discipline paid back during drafting: a fourth bullet (prompt-cache friendliness — Anthropic / OpenAI 90% read discount on cached `AGENTS.md`) was identified, evaluated, and **cut** to spike §3.4 fallback in the same hour the README was written, rather than leaking into the published copy and then having to be edited out later. Three claims survive: cost compounds per-query while extraction amortises (\$0.20–\$0.60/query for 100–300K-token re-shipping vs. 1–5K-token `AGENTS.md` once per session, two orders of magnitude); long-context retrieval still degrades on non-headline info past ~128–256K tokens (Liu et al. 2023 + Kuratov et al. 2024 BABILong, hyperlink-cited not inline-quoted, framed honestly as "long context works well for most-recent / most-prominent turns; degrades on the everyday 'what did we decide three weeks ago' question" rather than the false "long context is broken"); long context is per-machine while `AGENTS.md` is per-repo (re-states v2.4 positioning, cross-links to README §"What only ai-memory does"). The original sketch above mentioned "≈$3/query" as a headline figure; the spike replaced this with a defensible \$0.20–\$0.60/query band drawn from real session-token observations + provider pricing pages, because $3/query assumes a pathological 1M-token dump, and HN reviewers will catch unfalsifiable single numbers faster than they will an honest range. **Surfaces touched**: README.md §FAQ + README.zh-CN.md (mirror, register adjusted, code/tool names in English) + `docs/competitive-landscape.md` §"What recent HN launches teach us" item 5 (single back-link, no duplicated argument). **Re-spike triggers locked** for when this FAQ has to be revised: sub-\$0.50/M frontier pricing, <5% retrieval degradation past 500K, native cross-session compression in editors, provider prompt-cache pricing changes. **Honest gaps documented in spike §7**: no first-party Gemini retrieval measurement (rely on BABILong); "moderate session = 100–300K tokens" is from ~50 maintainer sessions, not community-wide; prompt-cache pricing is provider-specific and parked in §3.4. No new tests — pure docs change; test suite remains 578. Effort ~45 min total, well below the 0.5 dev-day estimate. The smallest-possible spike-first application (one section of one README) — proves the discipline scales down to single-paragraph docs changes, not just multi-day code work, which makes it cheap enough to apply to v2.5-10 (memory↔commit linking spike) and any future docs-only credibility surface.

#### v2.5-10 — Memory ↔ commit / file-path linking (spike only)

Each memory `.md` gets a metadata block `implemented_in: [<sha>, <path>]` that `ai-memory` populates by scanning `git log` for keyword/title overlap. Surfaces in `recall` and the dashboard.

**Why "spike only" in v2.5:** requires a similarity-scoring decision (naive substring vs embedding-based vs LLM-judged) and a UX decision (auto-link vs manual confirmation). A bad auto-linker is worse than no linker. Land the design doc this cycle, ship the feature in v2.6.

**Strategic value:** no competitor in any of our buckets can do this — they don't have plain-Markdown-in-git as their primary substrate. This is the kind of differentiator that earns a v2.6 flagship slot, not a v2.5 sub-feature.

**Est:** 1 dev day for the spike (design doc only). **Blocking:** nothing.

> **Update 2026-04-27 (post-spike):** Spike shipped same day as v2.5-09, immediately after the v2.5-09 audit pass. Spike-first discipline held a seventh consecutive time (after v2.5-04 / 05 / 06 / 07 / 08 / 09): [`docs/memory-commit-linking-spike-2026-04-27.md`](../memory-commit-linking-spike-2026-04-27.md) locks every decision the v2.6 implementer would otherwise have to make under deadline pressure. The discipline paid back **three times** during drafting: (1) original ROADMAP sketch `implemented_in: [<sha>, <path>]` flat-tuple schema would have shipped — schema couldn't represent multi-commit implementations or auto-vs-manual provenance; revised to `links.implementations[]` list with per-entry `confirmed_by: auto|manual` and idempotent `first_linked`; (2) "auto-link or nothing" single-threshold UX would have shipped — auto-link bad → silent damage to `recall` accuracy; revised to three-band model (auto / suggest / drop) with `--clear-auto` recovery; (3) LLM-judged scoring as a default would have surfaced as the obvious "best accuracy" choice — but cost scales O(memories × commits), and rejection criteria are exactly what humans run after seeing a candidate, so it's strictly worse as an automated layer than as the human's confirmation step. **Locked stack:** substring/weighted-Jaccard default (zero new deps) + embedding cosine opt-in via `--similarity embedding` (reuses existing `.embeddings.json` infra) + LLM-judged **rejected for v2.6 *and* v2.7**. **Locked schema:** `links.implementations[]` with §3.3 invariants. **Locked command surface:** `ai-memory link` with subcommands as flags, idempotent writes, bounded 10s `git` timeouts mirroring v2.4 `recall`. **`linking.enabled: false` opt-in default** in v2.6 — flipping ON in a minor would silently mutate every existing user's memory frontmatter on `extract` (v3.0 breaking-change vector; same discipline as v2.5-05 `--redact`). **v2.6 implementation plan broken into 10 ≤1-day chunks** (§4) totalling ~6 dev days incl. budgeted same-day audit pass + ground-truth corpus build on the ai-memory repo itself. **5 re-spike triggers + 6 known-unknowns documented.** **Honest assessment baked in (§10):** no competitor in any of our three buckets can structurally copy this — chat-history extractors lack commit linking, git-markdown runtimes lack memory files, opaque-DB runtimes have no memory-in-user's-git substrate. Linker turns "your memories are in git" from a *substrate* into a *substrate that compounds*. **Strategic priority for v2.6: flagship** — confirms the read this strategy ADR called out at the original sketch. v2.6 ROADMAP should lead with v2.5-10's implementation, not slot it into a corner. No code in `src/`; pure docs deliverable; test suite remains 578. Effort ~50 min, well under the 1 dev-day estimate — the "decisions locked early, alternatives evaluated for future re-visit" structure of all six prior spike docs gave a re-usable template, so the v2.5-10 spike is the longest of the seven (~470 lines because the design space spans 3 simultaneously-locked dimensions) but produced fastest. **Pattern lock-in for v2.6:** every feature touching frontmatter / git / external boundaries gets a spike doc before code, full stop. Cost is now a budgeted line item, not free overhead.

## Why these ten, ranked this way

1. **CCEB precision (01) is the only item that turns into republishable launch content.** F1 56 → 70 is HN-postable; "we added a 5th editor" is not. This dominates the ranking.
2. **`try` (02) closes a measurable conversion leak.** Every visitor who bounces at "needs OPENAI_API_KEY" is a permanent loss. The fix is half a day of work and re-uses an asset we already have.
3. **Distribution (03) compounds.** Every marketplace listing is a permanent passive surface. Doing it in week 1 means weeks 2–4 of v2.5 dev benefit from incoming traffic.
4. **Skills (04) is the strategic bet.** Hot in 2026-Q2, no competitor in any bucket has touched it, parallels the `AGENTS.md` play that worked. But it's a bet, not a known win — that's why it's #4, not #1.
5. **Redact (05) opens enterprise.** Cheap, fits a clear gap, low risk.
6. **Codex (06) is keeping-up work.** Important for "actively maintained" perception, not for differentiation.
7. **AGENTS.md eval (07) is the complementary benchmark.** Smaller cousin of CCEB, addresses a different skeptic.
8. **CCEB v1.1 + LongMemEval subset (08) defends future critiques.** Less urgent than 01 because the v2.4 baseline already gets us into the conversation.
9. **README FAQ (09) is a half-day ammunition stockpile** for HN comments. Easy.
10. **Memory linking (10)** is the v2.6 flagship preview. Spike-only this cycle.

## Alternatives considered

### A. Lead v2.5 with a VS Code extension (rejected)

A thin VS Code extension wrapping the CLI + dashboard. ~3–4 dev days, opens 10M-user marketplace exposure.

**Rejected because:**

- VS Code extension marketplace is high-volume, low-conversion compared to HN / Reddit for power-tool categories targeting other developers.
- Maintenance burden is permanent (VS Code API churn, marketplace policy compliance).
- The same exposure is achievable by being a great CLI + MCP server that VS Code users already discover via Cursor / Copilot integration.

Re-evaluate in v2.6 if CLI growth plateaus.

### B. Lead v2.5 with cross-file memory linking (rejected, kept as spike-only)

The "memory ↔ commit / file path" linking idea is genuinely differentiating and only we can do it (git-native plain-Markdown substrate). But:

- Requires a similarity-scoring decision that's non-trivial; bad auto-linking is worse than no linking.
- The narrative is harder to compress for HN — needs visuals to land, where the precision-lift story is one number.
- Better as a v2.6 flagship after v2.5 ships the proven-channel (Skills + precision lift) wins.

Kept as v2.5-10 spike-only to capture the design notes while context is fresh.

### C. Pause development and double-down on user research (rejected)

The case for: 30 days post-launch, do nothing technical except respond to issues, re-prioritise from real usage.

**Rejected because:** v2.4 was published today and HN promotion hasn't started, so "real usage" data won't accumulate fast enough to fill a 4-week pause; meanwhile competitors ship weekly and the Skills window won't wait.

The middle path is what this ADR proposes: ship items 01–03 in week 1 (which are user-flow improvements drawing on first-impression data we already have), then re-evaluate items 04–10 against post-launch issue traffic at the start of week 2.

### D. Cut the benchmark work, lead with features only (rejected)

The case for: we already published a CCEB number; benchmark work has diminishing returns vs. shipping new capabilities.

**Rejected because:** the over-extraction problem isn't a benchmark cosmetic — it directly degrades the AGENTS.md output every user reads. The benchmark is also our most-referenced asset by external commentary; refreshing it is content as much as engineering.

## Non-goals

- **No rebranding or repositioning.** ADR [2026-04-25-category-positioning.md](2026-04-25-category-positioning.md) holds. v2.5 is execution within the established positioning, not a re-pitch.
- **No new memory types.** The 5-type schema (decision / architecture / convention / todo / issue) is stable and load-bearing. Adding a 6th type would invalidate CCEB v1 and require a fixture rewrite.
