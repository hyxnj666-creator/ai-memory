# ADR 2026-04-26 — Post-v2.4 strategy & v2.5 scope

**Status:** Accepted
**Date:** 2026-04-26
**Owner:** @hyxnj666-creator (Conor Liu)
**Supersedes / extends:** [2026-04-25-category-positioning.md](2026-04-25-category-positioning.md) — category positioning is unchanged. This ADR settles **what we ship in v2.5 and in what order**, given v2.4 is now published and the 1000-star push window is open.

## Context

`ai-memory-cli@2.4.0` published to npm on 2026-04-26 (commit `3f21251`, tag `v2.4.0`). At publish:

- **Tests:** 431 across 25 files; typecheck clean; build clean.
- **Headline numbers:** CCEB v1 baseline F1 56.0% (P 43.8% / R 77.8%) on `gpt-4o-mini`, 9 fixtures.
- **Capabilities shipped this cycle:** AGENTS.md reverse-sync, `recall` git time-travel, `doctor` health check, dashboard polish (onclick escape + pagination at 50/page), CCEB framework + first published baseline, Cursor path display fix.
- **Outstanding from v2.4 plan:** hero GIF render (POSIX-only via `vhs`, deferred to a macOS/Linux machine; manual Windows fallback documented).
- **HN promotion:** decoupled from publish. Earliest target window Tue 2026-04-28 08:30 ET (= Beijing 20:30) per [`launch-plan.md`](../launch-plan.md).

The ADR [2026-04-25-category-positioning.md](2026-04-25-category-positioning.md) closed the "what slot do we claim" question. With v2.4 now in users' hands, the open questions are:

1. **What does v2.5 ship**, given the four "only we do this" claims have all landed code (#1 chat-history extraction, #2 AGENTS.md sync, #3 typed memory, #4 team-native)?
2. **Which 2026-Q2 market signals not yet in `competitive-landscape.md`** are worth chasing this cycle vs. parking for v2.6+?
3. **What's the single-biggest leverage** we have on the 30-day-after-launch metrics in [`launch-plan.md`](../launch-plan.md) (300-star floor, 1000-star stretch)?

## Decision

v2.5 scope is the following ten items, **stack-ranked by edge-per-day toward the 1000★ target**, not by feature size or technical interest. Items 01–03 are committed (must ship). Items 04–07 are strongly recommended, conditional on 01–03 landing on time. Items 08–10 are exploratory.

### Must ship (committed, week 1)

#### v2.5-01 — Lift CCEB precision from 43.8% to ≥60%, republish baseline

The over-extraction pattern is well-understood — see [`docs/benchmarks/cceb-baseline.md`](../benchmarks/cceb-baseline.md) §"Concrete v2.5 work this baseline points at". Three concrete changes:

1. **Prompt-side:** explicit "one decision per discussion thread" instruction; merge sub-claims into the parent's `reasoning` / `alternatives` / `impact` rather than as standalone memories.
2. **TODO discipline:** require explicit commitment language ("we will", "TODO:", "action item") before promoting a follow-up action to a standalone TODO memory. Default conservative; user can override per-extract.
3. **Type-classification calibration:** targeted few-shot for the convention-vs-decision boundary (cceb-002) and architecture-vs-decision boundary (cceb-003).

Re-run baseline against `gpt-4o-mini` on the same 9-fixture suite. Target: **F1 65%+ at recall ≥75%** — don't trade recall below ~75% even for a higher F1 headline; recall regressions are silent failures users won't catch.

**Why this is item #1:** the republished number is the single most reusable piece of v2.5 launch content. F1 56% → 70% is a *re-postable* HN narrative ("we listened, here's the new number") — one of the few ways to legitimately get a second front-page window. It's also the bottleneck for AGENTS.md output quality: if `extract` precision is 44%, the auto-generated `AGENTS.md` is half noise.

While we're in there, fix the cosmetic `openai (default)` model-label artefact called out in the baseline honesty notes.

**Est:** 2–3 dev days. **Blocking:** nothing.

#### v2.5-02 — `ai-memory try` (no-API-key demo mode)

Repurpose the existing [`docs/assets/demo/scenario/`](../assets/demo/scenario/) 3-memory hand-curated store as a runnable demo. Behaviour:

```bash
npx ai-memory-cli try
# - Bootstraps a tmp dir with the bundled scenario
# - Runs `rules --target agents-md` against it
# - Shows the generated AGENTS.md inline
# - No LLM call, no API key required
```

**Why:** today's first-run path requires `OPENAI_API_KEY` before the user sees any output. The npm-page → `npx ai-memory init` → "needs API key" → bounce funnel is the largest avoidable conversion leak. Both `cortexmem` and Palinode share this leak, so closing it is also a wedge.

**Implementation:** ship the existing `scenario/.ai-memory/` as part of the published tarball (add `docs/assets/demo/scenario/**` to `package.json` `files`), copy to a tmp dir on `try`, run the existing `rules` command path. No new pipelines.

**Est:** 0.5–1 dev day. **Blocking:** nothing.

#### v2.5-03 — MCP marketplace + GitHub Topics + AGENTS.md spec PR

One-time distribution work that pays back forever:

- Submit to [mcp.so](https://mcp.so/), [glama.ai/mcp/servers](https://glama.ai/mcp/servers), [cursor.directory/mcp](https://cursor.directory/mcp).
- Add GitHub Topics to the repo: `agents-md`, `cursor-rules`, `claude-code`, `chat-history`, `mcp-server` (verify no overlap with existing topics).
- Open a PR against the AGENTS.md spec repo (Linux Foundation Agentic AI Foundation) listing `ai-memory-cli` under "known generators / consumers".
- Submit to `awesome-mcp-servers`, `awesome-cursor`, `awesome-ai-coding`, `awesome-agents-md` (per existing `launch-plan.md`).

**Est:** 0.5 dev day. **Blocking:** v2.5-01 should land first so the marketplace listings cite the improved benchmark number, not the v2.4 baseline.

### Strongly recommended (target: ship if 01–03 land on time, weeks 2–3)

#### v2.5-04 — Anthropic Skills output (`rules --target skills`)

Anthropic Skills (introduced 2026-Q1) sit alongside `AGENTS.md` as the emerging cross-tool agent-instruction layer. They differ from `AGENTS.md` in two ways:

- Loaded **dynamically** by description matching, not always-on.
- Per-skill granularity (one file per capability) vs `AGENTS.md`'s single project-level rules document.

Natural mapping for `ai-memory`:

- One skill per (memory type × theme), e.g. `coding-conventions.skill`, `system-architecture.skill`, `decision-log.skill`.
- `description` field auto-generated from the constituent memory titles (concise "when to load" hint Claude uses to decide whether to pull the skill into context).
- Body composed from the relevant typed memories.

**Spike before commit:** validate the exact Skills schema against current Anthropic docs (the field set has been moving through 2026-Q1/Q2). Land a 1-page "Skills schema snapshot — date stamp" doc in `docs/` before implementing so future maintainers know which spec we targeted.

**Why fourth, not first:** `AGENTS.md` is the proven standard (60K+ adopters, Linux Foundation stewardship); Skills is a bet on the next 12 months. We should ship the bet, but only after the proven channel is humming. Also: Skills schema is still in flux; over-investing before the spec stabilises is wasted work.

**Why ship at all:** no competitor in any of our three buckets currently outputs Skills. Being first in this slot is the same play that worked for `AGENTS.md` reverse-sync in v2.4 — claim the unclaimed surface before the niche consolidates.

**Est:** 2–3 dev days (1 spike + 1–2 impl). **Blocking:** nothing technical; conditional on Skills spec being stable enough to target.

#### v2.5-05 — `--redact` flag for PII / secrets / internal URLs

Today `extract` sends conversation excerpts to the configured LLM provider. "Local-first" applies to *storage*, not to *extraction calls* — and the README is honest about this. A `--redact` flag (regex denylist for API keys, internal-domain URLs, email addresses) opens the "can I run this on a corp repo" door without changing the architecture.

Default rules to ship:

- `(sk|pk|aws|github_pat|ghp|xoxb)_[A-Za-z0-9]{20,}` — common token shapes.
- `https?://[^/]*\.(?:internal|corp|local|lan)/` — internal hostnames.
- RFC5322 emails.
- Optional: JWT-shaped tokens.

Custom rules via `.ai-memory/.config.json` `redact` array (regex strings; precompiled once per extract run).

**Why:** opens enterprise + safety-conscious users (HN niche, r/selfhosted, r/cscareerquestions). Cheap to ship, high signal-to-effort.

**Est:** 1 dev day. **Blocking:** nothing.

#### v2.5-06 — OpenAI Codex CLI source (5th editor)

OpenAI's `codex` CLI (released 2026-Q1, separate from codex.com) joins Cursor, Claude Code, Windsurf, Copilot Chat as the 5th supported editor. Conversation files live in `~/.codex/sessions/` (verify exact path during implementation).

Architecture: same shape as the Claude Code adapter — JSONL per session, source detection in `doctor`, source driver in `src/sources/codex/`. New tests follow the existing 4-source pattern (~10–15 unit tests).

**Why:** keeping pace with new tools is a low-effort, high-signal "the project is alive and current" indicator. Codex CLI users are a small but technical audience overlapping heavily with our target.

**Est:** 1–2 dev days. **Blocking:** v2.5-04 takes priority if there's contention (Skills is more strategic).

#### v2.5-07 — AGENTS.md downstream evaluation

CCEB measures **extraction quality** (chat → typed memory). We have no number for **rule-following quality** (memory → AGENTS.md → does the agent actually obey?). Methodology:

1. Drop an `ai-memory`-generated `AGENTS.md` into a controlled empty repo with 5 known conventions.
2. Define 10 micro-tasks that exercise specific rules (e.g. "add a function that fetches data from `/api/users`" against a "use cursor pagination, never offset" convention).
3. Run each task in Cursor and Claude Code; manually score "rule followed" (0/1) per observation.
4. Publish a single number: "X / 50 rule observations honored across Cursor + Claude Code."

This is a **complementary** credibility lever to CCEB — different layer, different question. CCEB shields against "extraction is unmeasured"; this shields against "the rules you generate are noise the agent ignores".

**Why this matters for HN:** the most common skeptical comment on tools in this niche is "AGENTS.md / Cursor Rules don't actually steer agents reliably". Having a number — even a small-N one — turns a vibe argument into a data argument.

**Est:** 1–2 dev days. **Blocking:** nothing technical; methodology sign-off needed before publishing.

### Optional / exploratory (weeks 3–4 if scope allows)

#### v2.5-08 — CCEB v1.1 (30 fixtures) + LongMemEval 50-query subset

Already queued for v2.5 in the v2.4 ROADMAP. 9 fixtures is small-N; 30 fixtures lets us claim per-type precision/recall with tighter error bars. The LongMemEval subset is the apples-to-apples shield against "but how do you compare to runtime memory benchmarks?"

**Est:** 3–4 dev days (mostly fixture authoring). **Blocking:** v2.5-01 (re-baseline once on v1, then fixture-grow on v1.1).

#### v2.5-09 — README FAQ: "Doesn't 1M-token context obsolete you?"

A single section, 1–2 paragraphs, pre-empting an HN question we know will come:

- At ~$3/1M input tokens, dumping 1M tokens of raw chat history per query is ≈$3/query — token economy *gets harder*, not easier, as context grows.
- Long context degrades on retrieval at the tail (well-documented needle-in-haystack failures); compressed typed memory outperforms 1M raw tokens on the dimensions we measure.
- `AGENTS.md` is loaded by **every** editor session, not one query — per-token cost compounds across sessions.

**Est:** 0.5 dev day. **Blocking:** nothing.

#### v2.5-10 — Memory ↔ commit / file-path linking (spike only)

Each memory `.md` gets a metadata block `implemented_in: [<sha>, <path>]` that `ai-memory` populates by scanning `git log` for keyword/title overlap. Surfaces in `recall` and the dashboard.

**Why "spike only" in v2.5:** requires a similarity-scoring decision (naive substring vs embedding-based vs LLM-judged) and a UX decision (auto-link vs manual confirmation). A bad auto-linker is worse than no linker. Land the design doc this cycle, ship the feature in v2.6.

**Strategic value:** no competitor in any of our buckets can do this — they don't have plain-Markdown-in-git as their primary substrate. This is the kind of differentiator that earns a v2.6 flagship slot, not a v2.5 sub-feature.

**Est:** 1 dev day for the spike (design doc only). **Blocking:** nothing.

## Why these ten, ranked this way

1. **CCEB precision (01) is the only item that turns into republishable launch content.** F1 56 → 70 is HN-postable; "we added a 5th editor" is not. This dominates the ranking.
2. **`try` (02) closes a measurable conversion leak.** Every visitor who bounces at "needs OPENAI_API_KEY" is a permanent loss. The fix is half a day of work and re-uses an asset we already have.
3. **Distribution (03) compounds.** Every marketplace listing is a permanent passive surface. Doing it in week 1 means weeks 2–4 of v2.5 dev benefit from incoming traffic.
4. **Skills (04) is the strategic bet.** Hot in 2026-Q2, no competitor in any bucket has touched it, parallels the `AGENTS.md` play that worked. But it's a bet, not a known win — that's why it's #4, not #1.
5. **Redact (05) opens enterprise.** Cheap, fits a clear gap, low risk.
6. **Codex (06) is keeping-up work.** Important for "actively maintained" perception, not for differentiation.
7. **AGENTS.md eval (07) is the complementary benchmark.** Smaller cousin of CCEB, addresses a different skeptic.
8. **CCEB v1.1 + LongMemEval subset (08) defends future critiques.** Less urgent than 01 because the v2.4 baseline already gets us into the conversation.
9. **README FAQ (09) is a half-day ammunition stockpile** for HN comments. Easy.
10. **Memory linking (10)** is the v2.6 flagship preview. Spike-only this cycle.

## Consequences

### Cadence target

3–4 weeks total:

| Week | Items | Output |
|---|---|---|
| 1 | 01, 02, 03 | `ai-memory-cli@2.5.0` patch with republished CCEB + `try` + distribution surfaces live |
| 2 | 04 (spike + impl), 05 | Skills bundle output + redact flag |
| 3 | 06, 07 | Codex source + AGENTS.md downstream eval |
| 4 | 08, 09, 10 | CCEB v1.1, FAQ section, memory-linking spike |

If 01 slips past week 1, descope 04 to v2.6 — better to ship a clean precision improvement than a mediocre Skills spike.

### Re-launch timing

A v2.5 release with republished CCEB + Skills support is a credible second HN post (≈4 weeks after v2.4 launch). Same posting window: Tue 08:30 ET. Title candidate: *"Show HN: ai-memory v2.5 — Skills + AGENTS.md auto-generated from your editor chat history (F1 56 → 70 on CCEB)"* — the precision lift in the title is the "we listened" signal HN rewards.

### What gets cut if launch metrics underperform

If 30-day metrics from the v2.4 launch fall below the [`launch-plan.md`](../launch-plan.md) "Success metrics" floors (300 stars / 100 weekly downloads / 5 stranger-opened issues), v2.5 scope freezes immediately. We re-read the launch plan, write a post-mortem ADR, and re-prioritise v2.5 against actual user feedback rather than this stack rank.

No "ship features anyway because they were planned".

### Risk to 千星 target

The dominant risk is now **positioning fatigue** — the four "only we do this" claims need fresh proof points every release, or they decay to background noise. v2.5-01 + v2.5-04 + v2.5-07 each generate one new proof point:

| Claim | v2.5 proof point |
|---|---|
| #1 zero `.remember()` | (existing, no new proof needed) |
| #2 native `AGENTS.md` | v2.5-07 — downstream eval shows generated rules actually steer agents |
| #3 typed memory | v2.5-01 — precision lift demonstrates per-type quality is improving |
| #4 team-native | (existing, no new proof needed) |
| **NEW** Skills + AGENTS.md dual output | v2.5-04 — first-mover on Skills generation |

A secondary risk is **Skills spec instability** — Anthropic has shipped breaking changes to Skills format twice in 2026-Q1 already. Mitigated by the spike-first approach and the date-stamped schema-snapshot doc.

## Alternatives considered

### A. Lead v2.5 with a VS Code extension (rejected)

A thin VS Code extension wrapping the CLI + dashboard. ~3–4 dev days, opens 10M-user marketplace exposure.

**Rejected because:**

- VS Code extension marketplace is high-volume, low-conversion compared to HN / Reddit for power-tool categories targeting other developers.
- Maintenance burden is permanent (VS Code API churn, marketplace policy compliance).
- The same exposure is achievable by being a great CLI + MCP server that VS Code users already discover via Cursor / Copilot integration.

Re-evaluate in v2.6 if CLI growth plateaus.

### B. Lead v2.5 with cross-file memory linking (rejected, kept as spike-only)

The "memory ↔ commit / file path" linking idea is genuinely differentiating and only we can do it (git-native plain-Markdown substrate). But:

- Requires a similarity-scoring decision that's non-trivial; bad auto-linking is worse than no linking.
- The narrative is harder to compress for HN — needs visuals to land, where the precision-lift story is one number.
- Better as a v2.6 flagship after v2.5 ships the proven-channel (Skills + precision lift) wins.

Kept as v2.5-10 spike-only to capture the design notes while context is fresh.

### C. Pause development and double-down on user research (rejected)

The case for: 30 days post-launch, do nothing technical except respond to issues, re-prioritise from real usage.

**Rejected because:** v2.4 was published today and HN promotion hasn't started, so "real usage" data won't accumulate fast enough to fill a 4-week pause; meanwhile competitors ship weekly and the Skills window won't wait.

The middle path is what this ADR proposes: ship items 01–03 in week 1 (which are user-flow improvements drawing on first-impression data we already have), then re-evaluate items 04–10 against post-launch issue traffic at the start of week 2.

### D. Cut the benchmark work, lead with features only (rejected)

The case for: we already published a CCEB number; benchmark work has diminishing returns vs. shipping new capabilities.

**Rejected because:** the over-extraction problem isn't a benchmark cosmetic — it directly degrades the AGENTS.md output every user reads. The benchmark is also our most-referenced asset by external commentary; refreshing it is content as much as engineering.

## Non-goals

- **No rebranding or repositioning.** ADR [2026-04-25-category-positioning.md](2026-04-25-category-positioning.md) holds. v2.5 is execution within the established positioning, not a re-pitch.
- **No new memory types.** The 5-type schema (decision / architecture / convention / todo / issue) is stable and load-bearing. Adding a 6th type would invalidate CCEB v1 and require a fixture rewrite.
- **No cloud sync.** Stays local-first. (Listed in `ROADMAP.md` Future Ideas; not a v2.5 candidate.)
- **No README hero rewrite.** v2.4's hero + four claims are the marketing surface for the v2.5 cycle too. Updates as proof points land are *sentence-level*, not *section-level*.
- **No multi-project knowledge sharing.** (Future Ideas; needs a separate ADR before scoping.)
